    # Week 5: Designing Blacksmith DB Coming Soon

    In-Class Exercise

    It's time to take the Blacksmith game to the next level, by moving the data from the session to a database. But before we can do that, we need to first design it.

    # Introduction

    Moving the Blacksmith game data to a database will give new abilities that are not possible with using session alone. For example, players will be able to continue game play even after the browser is close or play the same game across multiple browsers and devices. Players will also be able to have multiple different games all at the same time. But before we jump write into writing the PHP and SQL to make that happen, it is important that we plan out what the database should look like.

    Database Designing or Modelling is like creating a wireframe or mockup for a website. It gives the blueprint for database. This crucial with a database, because while it possible to make changes to a database after it has been deployed, it is HIGHLY discouraged. So it is important to carefully think about the problem and plan out all the needs for database before ever writing any code.

    # Understanding the Problem

    The database for Blacksmith will need to keep track of each game and all of the inventory items for each game. It should also have a record of each item that can be bought, made or sold as well as the requirements to complete the transaction to buy, make, and sell each item.

    From the description of the problem we can see that we will need at least 4 different tables for our Blacksmith database. Now, the columns of each table and the relationships between the tables can be worked out by creating a Database Model.

    # Creating a Database Model

    Creating a database model is actually pretty straight forward, and while there are plenty of options for database modelling software, like draw.io, a database model can just as easily done using a pen and paper.
